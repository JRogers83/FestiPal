# Festipals Code & UX Review
*Date: 2026-05-01 — updated 2026-04-29*

---

## Executive Summary

Festipals is a well-conceived, technically coherent app with a clean architecture: server components for initial data hydration, TanStack Query for optimistic mutations, a tidy Drizzle schema, and client-side clash detection. The foundation is solid and most of the developer work is done correctly. The visual design is strong — festival-grade aesthetic, consistent dark theme, and sensible component decomposition.

Three product-owner concerns raised since the first review expose genuine structural problems that go beyond polish.

**Multi-use invites (Point A):** The current schema allows exactly one redemption per token. A user sharing a link to a WhatsApp group chat of eight festival friends burns the token on the first person to tap Accept. This is not just a UX gap — it is a fundamental mismatch between how invite links are actually shared (broadcast to a group) and how the data model was designed (1:1 handshake). Fixing it requires a schema change and new application logic; a one-token-per-recipient workaround is not sufficient.

**Clash banner (Point B):** The banner across the top of the schedule duplicates information that is already communicated by dashed borders on individual act cards, but in a less readable, non-actionable form. It should be removed entirely. The grid's dashed-border treatment is the correct primary affordance.

**Mobile experience (Point C):** The grid is a desktop UI that has been allowed to scale down to mobile. At 375 px with 4 stages, each column is approximately 75 px wide, making act names unreadable and tap targets unreliable. The default view should be Artists on mobile, the grid should be an opt-in, and several layout decisions (tab bar overflow, panel as right-side drawer, header density) compound the problem. This section goes into specific detail.

The overall developer work quality has not changed since the first pass. All prior developer findings stand. The UX section has been substantially expanded below.

## Part 1: Developer Review

### Critical

#### 1.1 No input validation on colour field

**File:** `app/api/users/[userId]/route.ts`, lines 19–21

The PATCH endpoint accepts any string as a colour value and passes it directly to the database:

```ts
if (typeof body.colour === 'string') {
  data.colour = body.colour
}
```

Because this colour is later rendered as a CSS background via inline style in `ActCard`, `PersonRow`, and `ColourPicker`, an attacker who sends a crafted POST to `/api/users/[userId]` can store an arbitrary CSS string that executes in another user's browser when they view the combined schedule. The immediate risk is CSS injection (e.g. `expression()` in older engines, `url()` with data-URIs), and if the value ever reaches a `<style>` tag it becomes a vector for more.

**Fix:** Validate that the colour value is one of the 12 allowed hex strings from `USER_COLOURS`. In `app/api/users/[userId]/route.ts`:

```ts
import { USER_COLOURS } from '@/constants/colours'
const VALID_HEX = new Set(USER_COLOURS.map(c => c.hex))

if (typeof body.colour === 'string' && VALID_HEX.has(body.colour)) {
  data.colour = body.colour
}
```

The same guard should be added to `POST /api/users` if/when it accepts colour.

#### 1.2 UUID identity is never validated before use in API routes

**Files:** `app/api/selections/route.ts`, `app/api/connections/[id]/route.ts`, `app/api/invites/route.ts`

Several API routes accept `userId` from the request body and use it directly in DB queries without verifying it is a valid UUID format. An attacker can supply `userId: "'; DROP TABLE users; --"` (though Drizzle parameterises queries, so SQLi is not the concern) or, more relevantly, a value like `"../etc"` that causes unexpected behaviour in any path construction. More concretely, since `removeConnectionById` uses the `id` path parameter for both GET and DELETE (to fetch connections for a user AND to delete a specific connection), passing a user's UUID to DELETE would silently attempt to delete a record by that ID — which won't match any connection UUID, but the conflation of two operations under the same dynamic segment is a latent confusion risk.

**Note on the dual-use route:** `app/api/connections/[id]/route.ts` handles `GET /api/connections/[userId]` (fetch user's connections) and `DELETE /api/connections/[connectionId]` (delete a connection) on the same path. The spec split these into separate routes: `GET /api/connections/[userId]` and `DELETE /api/connections/[connectionId]`. Combining them is fine in practice, but it means there's no UUID-format check before either operation.

**Fix:** Add a UUID format guard at the top of any route that receives a UUID parameter from untrusted input:

```ts
const UUID_RE = /^[0-9a-f]{8}-[0-9a-f]{4}-[0-9a-f]{4}-[0-9a-f]{4}-[0-9a-f]{12}$/i
if (!UUID_RE.test(userId)) return NextResponse.json({ error: 'Invalid id' }, { status: 400 })
```

#### 1.3 No rate limiting implemented despite spec calling for it

**Spec reference:** Section 13, "Rate limiting on invite generation and selections via Vercel Edge Middleware"

There is no `middleware.ts` file in the project. The `POST /api/invites` endpoint has no limit, so a single user can generate thousands of invite tokens in seconds, polluting the `invite_tokens` table and potentially degrading performance. Similarly, `POST /api/selections` can be called in a rapid loop.

**Fix:** Implement a `middleware.ts` at the project root that applies IP-based rate limiting using `@vercel/edge-config` or a simple in-memory Map with a sliding window. At minimum, limit invite creation to ~10 per IP per minute and selection toggles to ~60 per IP per minute.

#### 1.4 Race condition in `redeemInvite`: check-then-act without a transaction

**File:** `lib/db/queries/invites.ts`, lines 29–51

The `redeemInvite` function reads the invite token, checks `usedAt`, then performs two separate writes (insert connection, update token). Under concurrent requests — e.g. two people simultaneously clicking "Accept" on the same shared invite link — both reads can return `usedAt = null`, both proceed past the guard, and both connections get inserted. The `onConflictDoNothing` on the `connections` insert makes the duplicate connection harmless, but both requests will mark the token used, and both will succeed with `{ success: true }`. This is probably acceptable in practice (the connection exists either way), but it is a correctness gap.

For multi-use tokens (see UX Point A below), this race condition is not just a correctness gap — it becomes a blocking problem. Multiple simultaneous redemptions of a multi-use token need to be correctly serialised against a use count.

**Fix (current schema):** Wrap the entire redeem operation in a single `UPDATE invite_tokens SET used_at = now(), used_by = $visitorId WHERE token = $token AND used_at IS NULL` and check `rowsAffected`. If 0, the token was already redeemed. This avoids the separate SELECT and removes the race.

**Fix (multi-use schema):** Use an atomic `UPDATE invite_tokens SET redemption_count = redemption_count + 1 WHERE token = $token AND (max_uses IS NULL OR redemption_count < max_uses) RETURNING *` and check whether a row was returned.

### Important

#### 1.5 `useUser` query always fires, even when `initialUser` has all required data

**File:** `app/u/[userId]/SchedulePageClient.tsx`, line 24; `lib/hooks/use-user.ts`

On every page load, `useUser(userId)` fires a GET to `/api/users/[userId]` even though `initialUser` is passed as a prop from the server component and contains the full user state. The query `staleTime` is 30 seconds, but on the first render the cache is empty so the request fires immediately regardless. This means every page load makes two round-trips to the DB for user data: one server-side (in `SchedulePageClient`) and one client-side (via `useUser`).

**Fix:** Pre-populate the TanStack Query cache with the `initialUser` data using `queryClient.setQueryData` in a `useEffect` or pass `initialData` to the query:

```ts
return useQuery<UserWithSelections | null>({
  queryKey: ['user', userId],
  queryFn: ...,
  initialData: initialUser ?? undefined,
  staleTime: 30_000,
})
```

Or use `placeholderData` if you want the server data to be immediately replaced once the client fetch resolves.

#### 1.6 `getLineup()` makes 3 sequential DB calls via `Promise.all` but stages/days are never cached

**File:** `lib/db/queries/lineup.ts`; `app/u/[userId]/page.tsx`

`getLineup()` correctly uses `Promise.all` for parallel queries. However, it is called on every server render of `/u/[userId]` — which happens on every day-tab change (since `?day=` triggers a full server render). The lineup data is static and could be cached via Next.js `unstable_cache` or the `revalidate` export, saving 3 DB round-trips on every subsequent visit.

**Fix:** Wrap `getLineup` with `unstable_cache`:

```ts
import { unstable_cache } from 'next/cache'

export const getLineup = unstable_cache(
  async () => { ... },
  ['lineup'],
  { revalidate: 86400 }
)
```

The `/api/lineup` route already sets `revalidate = 86400` but the server-side page function does not benefit from that cache.

#### 1.7 `getUserById` is an N+1 pattern with two sequential queries

**File:** `lib/db/queries/users.ts`, lines 14–22

The function issues one query to get the user, then a second to get their selections. These could be a single JOIN or a parallel `Promise.all`. As written they execute sequentially (the second query waits for the first), which adds one extra round-trip per user fetch.

**Fix:** Use `Promise.all`:

```ts
const [rows, userSelections] = await Promise.all([
  db.select().from(users).where(eq(users.id, id)),
  db.select({ actId: selections.actId }).from(selections).where(eq(selections.userId, id)),
])
const [user] = rows
if (!user) return null
```

The comment in `connections.ts` ("2 queries instead of 2N") shows the team is aware of this pattern — apply the same fix to `getUserById`.

#### 1.8 Invite page creates a new user UUID even if the visitor already has one

**File:** `app/invite/[token]/page.tsx`, lines 42–51

The `accept()` server action always calls `randomUUID()` and `createUser(visitorId)`, ignoring whether the visitor already has a Festipals URL saved. The spec (Section 6) explicitly states: "If visitor has a userId in their URL: use existing UUID." The current implementation breaks this: a returning user who clicks an invite link will get a new anonymous profile, losing all their previous act selections.

There is no mechanism for the visitor to pass their existing userId to this page. The original spec proposed a `?userId=` query param approach; the current implementation omits this entirely.

**Fix:** Add a `userId` query parameter input to the invite page. If the visitor has a saved Festipals URL, show a text field or link where they can enter/paste it. On the server action, use that ID if provided and valid (and exists in the DB), otherwise generate a new one. Alternatively, document explicitly in the UI that clicking Accept creates a fresh profile, and recommend copying their old schedule link.

#### 1.9 `targetNickname` variable assigned but never used in `direct/route.ts`

**File:** `app/api/connections/direct/route.ts`, line 42

```ts
targetNickname = target.nickname
```

This variable is declared with `let targetNickname: string` (implicitly, since TypeScript would infer it) but the branch at line 42 assigns to it without `let`/`const`, and it is never referenced afterward. The route fetches the target user again at line 49 (`const connected = await getUserById(targetId)`) to get the nickname for the response. This is an unnecessary double DB lookup: one at line 38 (`getUserById(targetUserId)`) and another at line 49. The intermediate assignment is dead code.

**Fix:** Remove the `targetNickname` assignment and use the `target` object already fetched at line 38 for the response nickname, only falling back to the second fetch when connecting via token.

#### 1.10 `useToggleSelection` optimistic update does not handle the case where `userData` is `null`

**File:** `lib/hooks/use-selections.ts`, lines 17–28

The `onMutate` handler calls `qc.getQueryData<{ selections: string[] }>(['user', userId])`. If the user has never been fetched (i.e. they're a brand new user whose first action is selecting an act), this returns `undefined` and the optimistic update is silently skipped. The act will still be selected server-side, but the UI won't reflect it until the `onSettled` invalidation fires and the refetch completes — causing a visible flash where the card appears to de-select then re-select.

**Fix:** When `old` is undefined, seed the cache with a minimal user object rather than returning `undefined`:

```ts
if (!old) return { id: userId, selections: [actId], nickname: 'Anonymous', colour: '#3b82f6', createdAt: '', lastSeen: '' }
```

Or initialise the cache earlier in the component lifecycle using `useEnsureUser`.

#### 1.11 `clash-detection.ts` has O(n²×m²) complexity with no guard

**File:** `lib/clash-detection.ts`, lines 20–38

The detection algorithm is four nested loops: users × users × selectionsA × selectionsB. For a group of 5 people each with 20 selections, this is 5×4/2 × 20 × 20 = 4,000 iterations — acceptable. But for a larger festival or a generous user, this could grow. More importantly, clashes are only filtered by `festivalDayId` inside `actsOverlap`, but `detectClashes` is passed `dayActs` (only the current day's acts) via the `acts` parameter while comparing against `ua.selections` which contains actIds from ALL days. This means `actMap` only contains day-scoped acts, so cross-day actIds simply return `undefined` from `actMap.get()` and are skipped — correct behaviour, but it means the selections arrays are iterated in full even though most will miss the map.

**Fix:** Pre-filter selections to only those present in `actMap` before the inner loops:

```ts
const userDaySelections = users.map(u => u.selections.filter(id => actMap.has(id)))
```

This reduces inner loop iterations significantly and makes the intent clearer.

#### 1.12 `shared` state in Header never resets if the user creates another invite

**File:** `components/Header.tsx`, lines 16, 31–55

Once `handleInvite` succeeds and `setShared(true)` is called, the button label permanently shows `✓ Shared` for the duration of the session, even if the user later wants to invite someone else. There is no timeout or reset mechanism.

**Fix:** Either reset `shared` to `false` after ~5 seconds, or reset it when `createInvite.isPending` becomes true again on a subsequent click:

```ts
useEffect(() => {
  if (createInvite.isPending) setShared(false)
}, [createInvite.isPending])
```

### Minor / Suggestions

#### 1.13 Dual CSS variable naming systems

**File:** `app/globals.css`

The codebase uses two parallel colour token systems: `--colour-*` (kebab-case British spelling) used everywhere in components, and `--color-brand-*` (camelCase American spelling via `@theme inline`) used nowhere outside this file. The `@theme inline` block is dead configuration carried over from the shadcn/ui initialisation. It adds confusion without value.

**Suggestion:** Remove the `--color-brand-*` entries from `@theme inline` since no component references them. Keep only the `--colour-*` tokens.

#### 1.14 `app/globals.css` has full shadcn/ui light/dark theme CSS variables that will never be used

The shadcn boilerplate `--background`, `--foreground`, `--primary`, etc. in `:root` and `.dark` are present (lines 80–146) but the app uses `--colour-*` tokens for everything. This is ~80 lines of dead CSS shipped in every page load.

**Suggestion:** Remove the shadcn `:root` and `.dark` variable blocks. The `@import "shadcn/tailwind.css"` may also be safe to remove if no shadcn components are actually used in the final UI.

#### 1.15 `connections/[id]/route.ts` name conflicts with spec

**File:** `app/api/connections/[id]/route.ts`

The spec defined this as `connections/[userId]/route.ts` (GET) and `connections/[connectionId]/route.ts` (DELETE). The implementation correctly merged them under `[id]`, but the parameter name `id` is ambiguous — it serves as both a userId and a connectionId depending on the HTTP method. A future developer reading the GET handler might be confused.

**Suggestion:** Rename the file to `[connectionId]/route.ts` and handle the semantic duality with a comment, or split into two separate route files per the spec.

#### 1.16 `createUser` in `POST /api/users` can return `null` but the route returns it directly

**File:** `app/api/users/route.ts`, line 12; `lib/db/queries/users.ts`, line 8

`createUser` uses `.onConflictDoNothing().returning()`. If the UUID already exists, `returning()` returns an empty array, so `user` is `undefined`, and `createUser` returns `null`. The route then serialises `null` as a 201 JSON response, which is incorrect — the client expects a user object. The `use-user.ts` `useEnsureUser` hook handles `409` but this case would return `201` with a `null` body.

**Fix:** After `createUser`, if `null` is returned (user already existed), fetch and return the existing user, or return `204 No Content` to indicate "already exists, no body."

#### 1.17 `adjustedEndMinutes` boundary condition: acts ending exactly at midnight

**File:** `lib/time.ts`, line 11

```ts
return end <= start ? end + 1440 : end
```

An act with `startTime: "00:00"` and `endTime: "00:00"` (a hypothetical zero-duration act) would trigger the midnight-span correction and return `0 + 1440 = 1440`, making it appear as a 24-hour act. While this edge case is unlikely in a real lineup, it suggests the condition should be `end < start` rather than `end <= start` for strict correctness.

**Suggestion:** Change to `end < start` to correctly handle the degenerate equal-times case.

#### 1.18 No `vitest.setup.ts` in the project root

The `vitest.config.ts` references `'./vitest.setup.ts'` but there is no such file visible in the project structure (not found in glob). If this file is missing, tests that use `expect(...).toBeInTheDocument()` from `@testing-library/jest-dom` will fail with "matcher not found" errors.

**Action required:** Verify `vitest.setup.ts` exists and contains `import '@testing-library/jest-dom'`.

#### 1.19 `getLineup` does not order acts

**File:** `lib/db/queries/lineup.ts`, line 9

```ts
db.select().from(acts),
```

Acts are returned in undefined order. The schedule grid positions them correctly by time calculation, but the `ArtistsList` component sorts them client-side on every render, which is correct but unnecessary if the initial order were consistent. This is a minor inefficiency.

**Suggestion:** Add `.orderBy(asc(acts.festivalDayId), asc(acts.startTime))` to give a predictable initial order.

#### 1.20 Test coverage is extremely thin

Only two test files exist for the entire application: `components/schedule/__tests__/ActCard.test.tsx` (5 tests) and `constants/__tests__/colours.test.ts` (4 tests). No tests exist for:

- `lib/clash-detection.ts` — the most complex pure function, with many edge cases (midnight-spanning acts, same-stage acts, cross-day selections)
- `lib/time.ts` — `adjustedEndMinutes` and `dayBounds` with midnight-crossing acts
- `lib/db/queries/*` — any database operation
- API routes — no integration tests
- `PeoplePanel` — the most complex UI component
- `parseConnectionInput` — the URL parsing function in `PeoplePanel.tsx` which handles multiple URL formats

**Suggestion:** Prioritise tests for `clash-detection.ts` and `time.ts` as they are pure functions with no external dependencies and directly affect correctness. The `parseConnectionInput` function is a good candidate for unit tests given its regex complexity.

## Part 2: UX Review

### Product Owner Feedback

This section addresses three issues raised directly by the product owner. Each is treated as a design problem requiring a concrete resolution, not just a flag.

---

#### Point A — Multi-use invite links: "What happens if the invite is shared to a group?"

**The problem in full**

The current `invite_tokens` table has a single `usedAt` timestamp and a single `usedBy` UUID:

```sql
token      uuid PRIMARY KEY DEFAULT gen_random_uuid()
created_by uuid NOT NULL REFERENCES users(id)
created_at timestamptz NOT NULL DEFAULT now()
used_at    timestamptz
used_by    uuid REFERENCES users(id)
```

The redemption logic in `lib/db/queries/invites.ts` checks `if (invite.usedAt) return { success: false, reason: 'already_used' }`. This means the first person to click Accept burns the token for everyone else in the group chat.

The natural sharing behaviour in practice is exactly the pattern this breaks: "here's a group of eight festival friends on WhatsApp — share the link to the chat." Person 1 clicks. Token gone. Persons 2–8 get "This invite link has already been used. Ask your friend to send you a new one." That error message is correct but the outcome is terrible — the inviter now has to generate seven more single-use tokens one at a time, each requiring a separate click of `+ Invite` and a separate paste into the chat.

The link-preview issue is now resolved (GET is safe since redemption requires a POST button click), but the single-use model is still wrong for group sharing.

**Proposed fix: multi-use tokens with optional cap**

This requires a schema change. The minimal viable change is:

```sql
-- Replace the single usedAt/usedBy pair with a max_uses limit and a counter
ALTER TABLE invite_tokens
  ADD COLUMN max_uses    smallint,       -- NULL = unlimited
  ADD COLUMN use_count   smallint NOT NULL DEFAULT 0,
  ADD COLUMN expires_at  timestamptz;    -- optional TTL, e.g. 7 days
```

A separate `invite_redemptions` table records each individual redemption:

```sql
CREATE TABLE invite_redemptions (
  id         uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  token      uuid NOT NULL REFERENCES invite_tokens(token) ON DELETE CASCADE,
  redeemed_by uuid NOT NULL REFERENCES users(id),
  redeemed_at timestamptz NOT NULL DEFAULT now()
);
```

The `redeemInvite` function becomes:

```ts
// Atomic increment with guard
const updated = await db
  .update(inviteTokens)
  .set({ useCount: sql`use_count + 1` })
  .where(
    and(
      eq(inviteTokens.token, token),
      or(isNull(inviteTokens.maxUses), lt(inviteTokens.useCount, inviteTokens.maxUses)),
      or(isNull(inviteTokens.expiresAt), gt(inviteTokens.expiresAt, new Date()))
    )
  )
  .returning()

if (updated.length === 0) return { success: false, reason: 'expired_or_full' }

// Record this redemption
await db.insert(inviteRedemptions).values({ token, redeemedBy: visitorId })

// Create connection
const [userA, userB] = [invite.createdBy, visitorId].sort() as [string, string]
await db.insert(connections).values({ userA, userB }).onConflictDoNothing()

return { success: true }
```

**The `already_used` error page** should be replaced with a page that says: "This link has been used the maximum number of times. Ask [creatorName] to send you a fresh invite link." — not shown at all if the token is unlimited-use.

**Default behaviour change:** When `+ Invite` is clicked, generate a token with `max_uses = null` (unlimited) and `expires_at = now() + 7 days`. This makes the default experience "share to a group, it just works" without requiring any UI change from the user. If the product later wants to limit, add a UI control.

**Impact on the `usedAt`/`usedBy` columns:** Both can be dropped from `invite_tokens` once the `invite_redemptions` table exists. Migration path: add new columns → deploy new code → drop old columns.

**Error message on the invite page (`app/invite/[token]/page.tsx`):** Currently the server-rendered page checks `invite.usedAt` and shows a hard error. Under the new model, a used-up or expired token should check `use_count >= max_uses` (for capped tokens) or `expires_at < now()`. The error message should vary accordingly: "This group invite link has expired" vs "This group invite has reached its limit."

**One additional issue this reveals:** The `InviteError` component renders a "Go to the homepage" link, not "Go back and get a new link" — the user's friend who sent the invite is more immediately useful than the landing page. Consider "Ask [creatorName] to send you a new link" as the error CTA.

#### Point B — Clash banner: remove it

**Verdict: remove the banner entirely.**

Looking at `components/schedule/ClashBanner.tsx` and `components/schedule/ScheduleGrid.tsx` together, the grid already communicates clash information through dashed coloured borders on `ActCard` elements:

```ts
const borderStyle = isClashing && clashColour
  ? { border: `2px dashed ${clashColour}` }
  : { border: '1px solid var(--colour-border)' }
```

The dashed border uses the clashing friend's colour, which is both more specific (you can see which friend's colour it is) and spatially located (it's right on the act that has the clash). The grid's treatment is the correct primary affordance. The banner is worse at the same job:

- It shows at most 3 clashes with a "+N more" overflow, so with more than 3 clashes the information is incomplete
- The "personA vs personB at 21:30" format doesn't tell you which act is clashing, only the time
- It occupies persistent vertical space above the grid on every page render when any clashes exist, pushing the act grid down — on a phone this is meaningful pixel loss
- It doesn't link to the clashing act (jump links were noted as missing in the previous review and remain absent)
- The grid already shows the dashed border at the exact position of the clash — the user's eye goes directly to it while scrolling

The only thing the banner does that the grid doesn't is give a count ("2 clashes"). This count has zero navigational or decision-making value on its own. If a user wants to understand their clashes, they scan the grid and see the dashed cards.

**What to do instead:** Remove `<ClashBanner clashPairs={clashPairs} />` from `SchedulePageClient.tsx`. Remove `ClashBanner.tsx`. The `clashPairs` prop on `ScheduleGrid` and the `clashingActIds` / `clashColourMap` computation already handle the visual signal correctly.

If there is a desire to surface clash count somewhere, a small badge on the "Group" panel handle (e.g. a red dot or number) when clashes exist would be both compact and useful — it tells the user "something requires attention in the group view" without consuming layout space in the main content area.

### Critical UX Issues (blocking or confusing flows)

#### 2.1 No onboarding: the user arrives at a blank schedule page with no instructions

**What the user sees:** After clicking "Build your schedule" on the landing page, they are immediately dropped into `/u/[uuid]` — a full schedule grid with stage headers, time slots, and a toolbar. The page looks functional but has no explanation of what to do next.

**What the user expects:** Some kind of "here's what you do" context. Most users have never used a festival planner without an account. They see a complex grid, don't know if anything is "loaded," don't understand how to interact with it, and may not even realise they can click on acts.

**What actually happens:** Nothing. The grid waits silently. Users who don't click on an act will not discover the core functionality. Users who want to invite friends will not know to look in the hidden sidebar.

**The specific failure:** There is no empty-state call to action, no tooltip, no first-run banner, and the header merely shows "Anonymous" with a colour dot — nothing explains what these are for.

**Fix:** Add a first-run banner (dismissible, stored in `sessionStorage` so it doesn't re-appear) that displays:
1. "Tap any act to add it to your schedule"
2. "Open the Group panel (right side) to invite friends"
3. "Bookmark this URL — it's your personal schedule link"

Alternatively, implement a minimal 3-step coach mark overlay on first visit.

#### 2.2 The people panel is invisible: the "Group" handle is not discoverable

**What the user sees:** A schedule grid that fills most of the screen. On the right edge, a narrow 36×96px tab with a `‹` chevron and vertical text "Group." On mobile, this is easy to miss. On desktop, it blends into the dark theme.

**What the user expects:** If they're thinking "I want to share this with my friends," they will look in the header (where the `+ Invite` button is), not at a barely-visible edge tab. The panel contains both "invite" functionality (via Manage > share link) and "group view" functionality, but neither of these use cases is advertised anywhere obvious.

**What actually happens:** Most users never open the panel. The `+ Invite` button in the header creates an invite token and either triggers the native share sheet or copies to clipboard — but this is a one-time single-use link, and after sending it, the friend's schedule is only visible once the friend accepts and both are connected. The panel is necessary to see this connection, but nothing tells the user to open it.

**The asymmetry:** On desktop, the panel opens by default (`useEffect` sets `open = true` when `window.innerWidth >= 768`), but on mobile it is closed. Mobile is the primary use case for a festival app.

**Fix:**
1. Make the panel visible by default on first load for all viewport sizes, at least on first visit.
2. After a successful invite is created (the `shared` state becomes `true`), show a prompt: "Invite sent! Check your Group panel to see when they join."
3. Increase the edge handle width and make the "Group" text horizontal, not vertical.

#### 2.3 Invite flow semantics are unclear: single-use vs. ongoing group

**What the user sees:** They click `+ Invite`, a link is copied. They send it to friend A. Later they want to invite friend B.

**What the user expects:** That they can share the same link, or that there's a persistent "group invite link" concept.

**What actually happens:** The second time they click `+ Invite`, a new token is generated and a new link is copied. If they send the old link to friend B, it says "already been used." There's no UI affordance that explains:
- Each click creates a brand new, single-use link
- The previous link is now invalid
- They can invite as many people as they want, but each needs their own fresh link

The `+ Invite` button also permanently changes to `✓ Shared` after the first use (see Issue 1.12), which makes it look like the action is done and the button is disabled.

If Point A is resolved (multi-use tokens), this issue largely disappears — the user creates one link, shares it anywhere, done. Until then:

**Fix:**
1. After copy/share, show a tooltip or toast: "Link copied — this link is single-use. Generate another to invite more friends."
2. Reset the button label to `+ Invite` after 3–4 seconds.
3. Consider adding a persistent "active invite link" concept: one unexpired token per user that can be regenerated on demand.

#### 2.4 No visual affordance that acts are clickable

**What the user sees:** A grid of act cards. They appear as styled blocks with a name and time. Nothing about them communicates "click me."

**What the user expects:** Some visual cue — a cursor change, a hover state, a checkbox — that indicates interactivity.

**What actually happens:** The cards are `<button>` elements so they will show a pointer cursor on desktop hover. But on mobile there is no hover state, and nothing communicates the tap-to-select interaction model. An unselected card and a selected card look like the same element in different colour states — but without already knowing the system, the user doesn't know what triggers the state change.

The `ArtistsList` view handles this correctly — it has an explicit selection circle on every row that communicates "tap to select." The grid has no equivalent.

**Fix:**
1. Add a subtle selection circle or checkbox icon in the corner of each unselected act card (similar to how `ArtistsList` uses a circle indicator).
2. Add a hover/focus background lift on desktop.
3. Consider adding a very brief micro-animation (scale 1.02 on click) so the user gets physical confirmation their tap was registered.

#### 2.5 "Grid" and "Artists" view toggle is ambiguous and easy to miss

**File:** `app/u/[userId]/SchedulePageClient.tsx`, lines 94–131

**What the user sees:** Two small buttons labelled "GRID" and "ARTISTS" in the top-right corner of the tab bar, styled in `font-size: 13px` uppercase. On mobile these are very small and hidden at the far end of the tab bar.

**What the user expects:** Given no prior context, "Artists" sounds like a list of artists (correct), but "Grid" is a technical term. Many users will not know what "Grid" means in this context. Additionally, on mobile, the tab bar can overflow horizontally (scrollable), so the view toggle may be scrolled out of sight.

**Fix:**
1. Use icons alongside the labels: a grid icon for Grid and a list icon for Artists.
2. Consider moving the view toggle to a more prominent position — perhaps a floating action button or a dedicated control row below the day tabs.
3. Label them more descriptively: "Timeline" and "List" are more universally understood than "Grid" and "Artists."

#### 2.6 Day tabs show only the day name, not the date — confusing for first-time users

**File:** `components/DayTabs.tsx`, line 35

```ts
{day.label.split(' ')[0]}
```

This strips everything after the first space, showing "Friday," "Saturday," "Sunday" without the dates. A user arriving in May/June who doesn't know the festival dates won't know which tab to activate.

**Fix:** Show the full label (e.g. "Friday 12 June") or at minimum "Fri 12 Jun." On small viewports, abbreviate to "Fri 12" but keep both pieces of information.

### Mobile UX Audit

The app was built quickly and it shows on mobile. This section documents every specific issue found and proposes concrete resolutions. The overarching problem is that the grid is a fixed-width desktop UI that has been allowed to scale down rather than adapt.

#### M-1 The grid is essentially unusable at 375 px with four or more stages

**What the user sees:** At 375 px (iPhone SE, iPhone 12/13/14 standard width), the grid has a time axis (~50 px) plus four stage columns at `minWidth: 140`. Total minimum grid width is 50 + 4×140 = 610 px. The `overflow-x-auto` wrapper allows horizontal scrolling, but each column at 140 px is already narrow — at narrower viewport widths the viewport itself clips the grid, so the user must scroll horizontally just to see stage 3 and 4.

**What the user expects:** To see the full schedule on screen without needing to scroll in two dimensions simultaneously. Vertical scrolling for time is expected. Horizontal scrolling to reveal hidden stages is a significant usability tax.

**What actually happens:** Act names at `text-xs` (12px) in 140 px columns are readable but compressed. With the time axis, only ~2.5 columns are visible at once on a 375 px viewport — meaning stages 3 and 4 require a deliberate horizontal swipe. A user wanting to compare stage 1 and stage 4 simultaneously cannot do so.

**Fix options:**
1. **Preferred: default to Artists view on mobile.** The `ArtistsList` component is single-column, scrollable, sortable, and shows all selection and clash information correctly. It is the right default for phones. In `SchedulePageClient.tsx`, initialise `view` based on viewport: `useState<'schedule' | 'artists'>(() => typeof window !== 'undefined' && window.innerWidth < 768 ? 'artists' : 'schedule')`. This is a one-line change that dramatically improves mobile first use.
2. **Secondary: add a horizontal stage selector for mobile.** Instead of showing all stages at once, show one stage at a time with left/right swipe navigation. This is a larger change but matches how most festival apps handle multi-stage on mobile.
3. **Do not reduce `minWidth` below 140.** At 120 px or below, act names are regularly truncated and time text at the act's start time is unreadable.

#### M-2 The tab bar row is not usable on small screens

**What the user sees:** The tab bar contains day tabs (scrollable `flex-1 min-w-0` container) and the Grid/Artists toggle (`flexShrink: 0`, always visible). The day tabs container has `overflow-x: auto` and `scrollbarWidth: none`, so the tabs themselves scroll independently. However, on a 375 px viewport with three day tabs and the view toggle, the day tabs must scroll to show all three days, and the toggle is fixed to the right.

**What actually happens:** The "Artists" button is always visible. The day tabs are scrollable. This is acceptable but the scrollability of the day tabs is not signalled to the user — there's no overflow indicator, no partial clip, no scroll indicator dots. A user who can only see "Friday" and "Saturday" but not "Sunday" on screen may not know to scroll.

**Additionally:** The entire tab row is `height: auto` driven by `py-3` on the tab buttons. This is not a problem per se, but the row combined with the header (56 px) and the tab bar (~44 px) consumes approximately 100 px of a 667 px phone screen (iPhone SE) before any content — 15% of the viewport is chrome. After accounting for the browser URL bar on Safari iOS (approximately 50 px when visible), the user has about 515 px of usable content area.

**Fix:**
1. Reduce tab padding to `py-2` on mobile (use Tailwind responsive prefix `md:py-3`).
2. Add a subtle gradient fade on the right edge of the day tabs container to signal scrollability: `::after { content: ''; position: absolute; right: 0; ... background: linear-gradient(to right, transparent, var(--colour-surface)) }`.
3. Or move the Grid/Artists toggle to a dedicated row below the day tabs so the tabs have the full width on mobile.

#### M-3 The people panel is a right-side drawer on all viewports — this is wrong for mobile

**What the user sees on a phone:** The panel slides in from the right at 256 px wide. On a 375 px phone, this leaves 119 px for the schedule grid — which is narrower than a single stage column's `minWidth: 140`. The grid effectively becomes invisible while the panel is open.

**What the spec said:** Section 7.2, "People panel — collapsible sidebar on desktop, bottom sheet on mobile." This was not implemented.

**What the user expects:** On mobile, panels that overlay content come from the bottom (WhatsApp, Spotify, Google Maps all use bottom sheets for secondary content). A bottom sheet that takes 50–60% of screen height leaves the top half of the schedule visible, giving the user context about what the group panel refers to.

**Fix:** Implement a media query breakpoint at 768 px. Below 768 px, render the panel as `position: fixed; bottom: 0; left: 0; right: 0; height: 65vh; border-radius: 12px 12px 0 0` instead of a right-side drawer. The "Group" edge handle should become a bottom-anchored button (or a persistent pill at the bottom of the screen, similar to a tab bar). This is the most impactful single mobile UX fix available.

The `useEffect` that sets `open = true` on `window.innerWidth >= 768` already correctly keeps the panel closed by default on mobile — but the open state still renders as a right panel. Only the positioning needs to change.

#### M-4 Header is dense and touch targets are small

**What the user sees:** The header is `h-14` (56 px). It contains: the Festipals wordmark (`text-xl`), a `ColourPicker` trigger (a small colour swatch), the nickname button (`text-sm`, unspecified width), and the `+ Invite` button (`px-3 py-1.5 text-xs`). On a 375 px viewport these four elements compete for horizontal space.

**What actually happens:** The `+ Invite` button has `py-1.5` which gives it a height of approximately 30 px — below the 44 px minimum touch target recommended by Apple's HIG and Google's Material Design. The colour swatch (via `ColourPicker`) and the nickname button have no minimum height enforced and will be similarly small.

**Fix:**
1. Ensure `+ Invite` button has `min-height: 44px` on mobile, or increase to `py-2.5` and `px-4`.
2. Ensure the nickname button has `min-height: 44px` or wrap it in a touch target div.
3. On very small viewports (< 360 px), consider hiding the festival wordmark or shrinking it to make room.

#### M-5 Act cards in the grid have very small touch targets for short acts

**What the user sees:** Acts shorter than 45 minutes render at a small pixel height via `minutesToPx`. A 30-minute act at 4 px/minute = 120 px height is fine. But the `minutesToPx` constant is not documented — if it is set lower (e.g. 2 px/minute), a 30-minute act would be 60 px tall. Short DJ sets or opening acts may be even smaller.

**What actually happens:** `ActCard` has no `minHeight` except for headliners (`headliner ? Math.max(height, 120) : height`). A 20-minute act could render as a 40–80 px tall button depending on the px/minute constant. Below 44 px, this is below minimum touch target. A 15-minute changeover act could render at 30 px — very difficult to tap accurately.

**Fix:** Add `minHeight: 44` (or `minHeight: '44px'`) to all `ActCard` buttons, not just headliners. Alternatively, increase the px-per-minute constant on mobile only.

#### M-6 The Artists list is significantly better on mobile and should be the default

This is already stated under M-1, but worth emphasising as a standalone finding. The `ArtistsList` component:
- Is fully single-column, no horizontal scrolling
- Has 44 px+ row heights (11px padding each side on a ~22px content row = ~44px touch targets)
- Shows a clear selection circle affordance
- Has sticky controls bar with day scope and sort controls
- Shows friend colour dots and stage/time metadata inline
- Supports sorting by time, name, or stage

The grid has none of these mobile-friendly properties. **The Artists list is a better mobile interface for building and reviewing a schedule.** The current design treats it as a secondary view accessible via a small toggle button; on mobile it should be the primary view.

#### M-7 No keyboard-dismissible drawers, sheets, or modals

**What the user sees:** The colour picker opens as an absolutely-positioned dropdown. There is no `Escape` key handler to close it, no outside-click handler, and no focus trap. On mobile, once the picker is open, there is no obvious way to close it without making a selection — the user cannot tap outside and expect it to dismiss (unless the component handles this, which is not visible in the files reviewed).

**Fix:** Ensure the colour picker closes on outside click (add a backdrop div or use a `useOnClickOutside` hook). Add `Escape` key handling. Test on iOS Safari where tap events on non-interactive elements do not always fire `click` events without explicit `cursor: pointer` on the backdrop.

#### M-8 Safari iOS viewport height issues with `h-screen`

**What the user sees:** The top-level layout is `<div className="flex flex-col h-screen">`. On Safari iOS, `100vh` includes the browser chrome (URL bar, bottom navigation bar) — the actual usable viewport is smaller. This causes the schedule grid area to be taller than the visible screen, producing a double-scroll situation: the page scrolls (because of the browser chrome), and the grid scrolls within the fixed `h-screen` container.

**Fix:** Replace `h-screen` with `height: 100dvh` (dynamic viewport height), which is supported in all modern mobile browsers including Safari iOS 15.4+. In Tailwind this is `h-dvh` (available in Tailwind v3.4+). This makes the layout correctly fill the visible viewport height regardless of browser chrome state.

#### M-9 No swipe gesture support

**What the user sees:** Day tabs are navigated by tapping. There is no swipe-left/swipe-right gesture to change days.

**What the user expects:** On mobile, content that is paginated by a tab row is almost universally navigable by horizontal swipe. Most festival apps, Spotify, and calendar apps implement this.

**Fix:** Wrap the main content area in a touch-gesture handler. On `touchstart`/`touchend`, detect a horizontal swipe with sufficient velocity (> 50 px, < 500ms) and navigate to the next/previous day. This is a `useSwipe` hook, approximately 30 lines. The `router.push` in `DayTabs` already provides the navigation target — the swipe handler just needs to call the same function with the adjacent day's ID.

#### 2.13 No loading states for act selection toggle

**File:** `lib/hooks/use-selections.ts`; `components/schedule/ActCard.tsx`

When a user taps an act to select it, the optimistic update fires immediately — good. But if the server request fails (network error, DB error), the UI rolls back silently via `onError`. ## Appendix: Schema change for multi-use tokens (Point A)

For reference, the complete schema change needed to support multi-use invite tokens:

**Current schema (`lib/db/schema.ts`):**
```ts
export const inviteTokens = pgTable('invite_tokens', {
  token:     uuid('token').primaryKey().defaultRandom(),
  createdBy: uuid('created_by').notNull().references(() => users.id, { onDelete: 'cascade' }),
  createdAt: timestamp('created_at', { withTimezone: true }).notNull().defaultNow(),
  usedAt:    timestamp('used_at', { withTimezone: true }),
  usedBy:    uuid('used_by').references(() => users.id),
})
```

**Proposed schema:**
```ts
export const inviteTokens = pgTable('invite_tokens', {
  token:     uuid('token').primaryKey().defaultRandom(),
  createdBy: uuid('created_by').notNull().references(() => users.id, { onDelete: 'cascade' }),
  createdAt: timestamp('created_at', { withTimezone: true }).notNull().defaultNow(),
  expiresAt: timestamp('expires_at', { withTimezone: true }),    // null = never expires
  maxUses:   smallint('max_uses'),                               // null = unlimited
  useCount:  smallint('use_count').notNull().default(0),
})

export const inviteRedemptions = pgTable('invite_redemptions', {
  id:          uuid('id').primaryKey().defaultRandom(),
  token:       uuid('token').notNull().references(() => inviteTokens.token, { onDelete: 'cascade' }),
  redeemedBy:  uuid('redeemed_by').notNull().references(() => users.id),
  redeemedAt:  timestamp('redeemed_at', { withTimezone: true }).notNull().defaultNow(),
}, t => ({
  uniq: unique().on(t.token, t.redeemedBy),   // prevent double-redemption by same user
}))
```

The `uniq` constraint on `(token, redeemed_by)` prevents the same user redeeming the same token twice (e.g. if they click Accept on the same link from two devices). The atomic `UPDATE ... SET use_count = use_count + 1 WHERE use_count < max_uses` prevents over-redemption under concurrent requests without needing a database-level lock.

When creating an invite via `POST /api/invites`, set `expiresAt = new Date(Date.now() + 7 * 24 * 60 * 60 * 1000)` by default (7-day expiry). `maxUses` defaults to `null` (unlimited). This makes sharing to a group chat work correctly with no UI changes required.

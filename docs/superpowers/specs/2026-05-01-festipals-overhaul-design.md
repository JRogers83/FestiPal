## Group A ‚Äî Share Story, Clash Visibility, Desktop Layout

### A0 ¬∑ Schema migration: multi-use invite tokens

**Problem:** `invite_tokens.used_at` / `used_by` make tokens single-use. Sharing a link to a WhatsApp group means the first person to tap Accept burns the token; everyone else sees "already used."

**Change to `lib/db/schema.ts`:**
- Remove columns `usedAt` (`used_at TIMESTAMPTZ`) and `usedBy` (`used_by UUID`)
- Add column `expiresAt` (`expires_at TIMESTAMPTZ NULLABLE`)

```sql
-- Migration (drizzle-kit push)
ALTER TABLE invite_tokens
  DROP COLUMN used_at,
  DROP COLUMN used_by,
  ADD COLUMN expires_at TIMESTAMPTZ;
```

**Default expiry at creation:** `'2026-06-21'` (one week after festival end). `NULL` = never expires. Set in `POST /api/invites` when creating the token.

**New validity check** in `lib/db/queries/invites.ts`:
```typescript
// Old: if (invite.usedAt) return { success: false, reason: 'already_used' }
// New:
if (invite.expiresAt && invite.expiresAt < new Date()) {
  return { success: false, reason: 'expired' }
}
```

**Files affected:**
- `lib/db/schema.ts` ‚Äî drop columns, add `expiresAt`
- `lib/db/queries/invites.ts` ‚Äî update `redeemInvite`, remove `markInviteUsed`, add `RedeemResult` reason `'expired'`
- `app/api/invites/route.ts` ‚Äî set `expiresAt` on creation
- `app/api/connections/direct/route.ts` ‚Äî remove `markInviteUsed` call
- `app/invite/[token]/page.tsx` ‚Äî change "already used" message to "this link has expired"

### A2 ¬∑ Clash borders on both acts in a pair

**Problem (F-1):** `ScheduleGrid` only applies the clash border to one side of a clashing pair (the act belonging to the current user's opponent). The current user's own clashing act shows no signal. The 2 px dashed border is also visually weak.

**Fix in `components/schedule/ScheduleGrid.tsx`:**

Extend `clashingActIds` and `clashColourMap` to include **both** acts in every clash pair, not just the current user's side:

```typescript
// Current: only marks personA's act if personA === currentUserId
// New: mark both acts in every pair, each tinted with the other person's colour
const clashingActIds = useMemo(() => {
  const set = new Set<string>()
  for (const cp of clashPairs) {
    set.add(cp.personA.act.id)
    set.add(cp.personB.act.id)
  }
  return set
}, [clashPairs])

const clashColourMap = useMemo(() => {
  const map = new Map<string, string>()
  for (const cp of clashPairs) {
    map.set(cp.personA.act.id, cp.personB.colour)
    map.set(cp.personB.act.id, cp.personA.colour)
  }
  return map
}, [clashPairs])
```

**Fix in `components/schedule/ActCard.tsx`:** Replace the `border: 2px dashed` with `box-shadow: inset 0 0 0 3px <colour>` so it doesn't conflict with card fill. Add a `‚öÝ` glyph in the top-right corner when clashing:

```typescript
// In the button style, replace borderStyle:
boxShadow: isClashing && clashColour
  ? `inset 0 0 0 3px ${clashColour}`
  : undefined,
// Add glyph alongside act name when isClashing:
{isClashing && (
  <span style={{ position: 'absolute', top: 3, right: 4, fontSize: 9, opacity: 0.8 }}>‚öÝ</span>
)}
```

### B8 ¬∑ Swipe between days on mobile

Add a horizontal swipe gesture to the main content area (`<main>` in `SchedulePageClient`) using **JavaScript event listeners only** ‚Äî no CSS `touch-action` changes (those have caused scroll regressions previously):

```typescript
// On the <main> element:
onTouchStart={(e) => { touchStartX.current = e.touches[0].clientX; touchStartY.current = e.touches[0].clientY }}
onTouchEnd={(e) => {
  const dx = e.changedTouches[0].clientX - touchStartX.current
  const dy = Math.abs(e.changedTouches[0].clientY - touchStartY.current)
  if (Math.abs(dx) > 60 && dy < 30) {
    // advance or retreat day tab
    const days = lineup.festivalDays.filter(d => daysWithActs.has(d.id))
    const idx = days.findIndex(d => d.id === activeDay)
    const next = dx < 0 ? days[idx + 1] : days[idx - 1]
    if (next) router.push(`/u/${userId}?day=${next.id}`)
  }
}}
```

The gesture only fires if horizontal movement greatly exceeds vertical, preventing conflict with normal vertical scroll.

### C7 ¬∑ "Already have a profile?" on invite-accept page

**Problem (1.8):** Accepting an invite always creates a new UUID, wiping existing selections.

**Fix in `app/invite/[token]/page.tsx`:**

> *"Already using Festipals? Paste your link to join as your existing profile."*

The form has two server actions:

2. **Join as existing** ‚Äî validates the pasted URL, extracts the UUID, verifies the user exists, then creates a connection between the existing user and the inviter. Redirects to the existing user's `/u/[uuid]`.

The `accept-existing` server action:
```typescript
async function acceptAsExisting(formData: FormData) {
  'use server'
  const pastedUrl = formData.get('existingUrl') as string
  const match = pastedUrl.match(/\/u\/([0-9a-f-]{36})/i)
  if (!match) redirect(`/invite/${token}?error=invalid_url`)
  const existingUserId = match[1]
  const existingUser = await getUserById(existingUserId)
  if (!existingUser) redirect(`/invite/${token}?error=user_not_found`)
  if (existingUserId === invite.createdBy) redirect(`/invite/${token}?error=self_invite`)
  await createDirectConnection(invite.createdBy, existingUserId)
  redirect(`/u/${existingUserId}`)
}
```

### D4 ¬∑ Fix time formatter

**Problem (F-7):** DB returns `TIME` as `"HH:MM:SS"`. Clash display shows `"15:50:00"` instead of `"15:50"`. Clash time reports start of earlier act instead of the overlap window.

**Fix in `lib/clash-detection.ts`:** Add `overlapStart` to `ClashPair`:

```typescript
export type ClashPair = {
  personA: { userId: string; nickname: string; colour: string; act: Act }
  personB: { userId: string; nickname: string; colour: string; act: Act }
  overlapStart: string  // max(actA.startTime, actB.startTime), formatted HH:MM
}
```
